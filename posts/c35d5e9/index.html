[BUAA OS Lab3]进程与异常

又从 mips_init 开始

在这篇文章中，我们依旧按照 mips_init 函数的执行顺序来探讨进程和异常的内容。

mips_init 中关于 Lab3 的代码有如下内容：

env_init

这个函数首先初始化了两个链表：env_free_list 和 env_sched_list。它们分别是双向链表和尾链表，其定义和相关的链表宏可以在 include/queue.h 中找到，这里不再赘述。

两个链表中的元素类型都是一样的：struct Env，我们在 include/env.h 中找到它的定义如下：

这个结构体类型被称为进程控制块Process Control Block, PCB，其中保存了一个进程所拥有的资源。

为了集中注意力，我们只讲解和 Lab3 相关的成员。

  • 和 Lab2 的 struct Page 类似，我们看到 struct Env 中也有内置的构成链表用的结构体，但不只一个，而是两个，它们分别是 env_link 和 env_sched_link。

  • env_tf 也是一个结构体，用于保存进程的上下文，在发生进程调度或者陷入内核的时候，会将当前进程的上下文（寄存器的值、PC 寄存器中的地址等等）保存在这个结构体中，它定义在 include/trap.h 中。

  • env_id 是每个进程的唯一标识符。

  • env_parent_id 是父进程的进程 id 。

  • env_status 表示这个进程的状态，一共有三种可能性：

  • env_asid 是进程的 ASID，用于 TLB 中。

  • env_pgdir 是当前进程拥有的页目录的虚拟地址。

  • env_pri 表示进程的优先级。

现在让我们回过头来看看这两个链表：env_free_list 类似于 page_free_list，将所有空闲的 PCB 组织在一起；env_sched_list 则意味着组织了所有被调度的 PCB。

然后，我们先把所有的 PCB 全部插入空闲链表中，表示在初始情况下没有被调度的进程，我们要将 PCB 的 env_status 字段设置为 ENV_FREE。

接下来我们还做了一件事：使用 page_alloc 函数申请了一个物理页，将其转为虚拟地址给到 base_pgdir。这是什么？这是一个页目录的基地址。下一步我们要在这个基地址对应的两级页表结构中做段地址映射，将虚拟地址段 [va, va + size) 映射到物理地址段 [pa, pa + size)，这样做的目的是让程序可以通过访问用户地址空间来获取 pages 和 envs 两个数组的信息，有点类似于 Lab2 提到的自映射。我们用到了一个名叫 map_segment 的函数：

它基本上就是一个 page_insert，不断地将 va + i 和 pa + i 建立映射关系，并设定权限位（我们当然不希望用户能够随意更改内核空间的内容）。

env_create

我们首先看看这个函数的参数是什么。

首先 const void *binary 是一个二进制文件，代表我们要创建进程的程序，二进制文件数据数组的大小是 size_t size，优先级是 priority，对应 struct Env 的 env_pri 成员变量。

接着调用 env_setup_vm 函数初始化 PCB 的用户地址空间，也就是为进程控制块创建对应的二级页表，并构建页表自映射：

env_setup_vm

我们先理清楚一个概念：之前 env_init 给 base_pgdir 赋了一个值，让它指向一个页目录，这个页目录被叫做模版页目录。之后每次我们创建一个新的进程，进程都会有自己的页目录，而这些页目录就必须先从模版页目录那里复制一份过来。

再来看代码：申请一个物理页，这就是我们说的进程自己的页目录，用 e -> env_pgdir = (Pde *)page2kva(p) 来将页目录和进程绑定起来。

接下来我们把页目录中之前做了段映射的页目录项全部复制过来。回顾一下，我们在 env_init 中用 map_segment 函数做了两次段映射，虚拟地址一次从 UENVS 开始，一次从 UPAGES 开始。在 mmu.h 中我们查看 MIPS 内存布局得到：

我们直接复制 UTOP 到 UVPT 这个范围内的页目录项。但还没完，最后我们还得将页目录那个特殊的页目录项指向页目录本身的物理地址，这样就构建了自映射[2]。

现在我们回到 env_alloc 函数。在 env_setup_vm 后，我们还要对 PCB 进行一些初始化，这样才算真正申请了一个有效的 PCB。和 Lab3 相关的初始化内容包括 env_id, env_parent_id, env_asid。

ASID 是有限的（26 = 64），在所有 ASID 分配后，应该返回异常。

下一步设置 Trapframe 中一些寄存器的值。CP0 的 SR 寄存器各位的含义可以参考计组分类的博客，这里不再赘述。
要注意 $29 代表 sp 寄存器，按理说应该在栈顶 USTACKTOP。但为了给 main 函数的参数 int argc 和 char **argv 留出空间，我们需要减去这两个类型的大小。

最后，别忘了从 env_free_list 中移除 e，它已经是一个被调度的 PCB 了。

下一步我们调用 load_icode 函数加载 ELF 二进制文件，并将 e 移动到已被调度的 PCB 链表 env_sched_list 中：

load_icode

load_icode 首先调用了 elf_from 函数读取 ELF 文件信息，这个函数只是简单地进行数据类型转换，并检查 ELF 文件头，这里略过。

接着用了一个宏 ELF_FOREACH_PHDR_OFF 来遍历 ELF 文件的段头表，如果段的类型是 PT_LOAD，则说明对应的段需要加载到内存中，调用 elf_load_seg 函数实现。elf_load_seg 不是我们的重点。这里简单讲解一下：

段的内存布局比较复杂，需要加载到的虚拟地址 va，内存长度等等都有可能不是页对齐的，elf_load_seg 会正确处理这些地址的偏移。对于每个需要加载的页，它会用对齐后的 va 等信息来调用一个回调函数 map_page（在我们的实现中，这是 load_icode_mapper），这个函数会完成分配所需的物理页面并在页表中建立映射的任务。

load_icode_mapper

我们要想将数据加载到内存，第一步得申请一个物理页，用 page_alloc 来完成。
接下来，如果有需要拷贝的数据，将其拷贝到这个页中（记得加上偏移量 offset）。
最后调用 page_insert 将虚拟地址映射到物理页上。为了区别不同进程的相同虚拟地址，我们附加 ASID 信息。

最后，将 Trapframe 的 EPC 寄存器设置为程序的入口地址。

这样，进程就被我们加载到了它独特的虚拟地址空间中。但我们还没能让它运行起来，而让它运行起来正是我们后面的工作。
