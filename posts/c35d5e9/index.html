MIPS CPU 处理一个异常时大致要完成四项工作：

  1. 设置 EPC 指向从异常返回的地址。

  2. 设置 EXL 位，强制 CPU 进入内核态（行使更高级的特权）并禁止中断。

  3. 设置 Cause 寄存器，用于记录异常发生的原因。

  4. CPU 开始从异常入口位置取指，此后一切交给软件处理。

而这句“一切交给软件处理”，就是我们当前任务的开始。

为什么时钟中断如此特殊呢？

时间片轮转

MOS 中进程调度是依靠一种叫做时间片轮转算法实现的。即我们为每个进程分配一个时间段，称作它的时间片。在时间片内进程可以运行，超过时间片，如果进程还没运行结束，就挂起，然后让另一个进程运行。如果在时间片内该进程阻塞或者结束运行，就立刻让另一个进程运行。

可操作系统如何知道时间片是否结束呢？这就要依靠硬件计时器产生的时钟中断了。MOS 中，时间片的长度是用时钟中断来衡量的。我们会设定一个时钟中断间隔，每经过间隔的长度，就产生一次时钟中断。比如某进程的时间片长度是 200 时钟中断间隔，那么时钟中断发生 200 次后，时间片结束。

我们知道 CP0 中有 Timer，用来产生时钟中断。在正式开始之前，我们需要给它初始化。具体来说，有两个寄存器用来控制 Timer：CP0_COUNT 会按照某种频率自增，当其等于 CP0_COMPARE 时，立刻触发时钟中断。我们要做的就是先清零 CP0_COUNT，并设置 CP0_COMPARE 为我们想要的周期，这样就完成了初始化：

进程调度

让我们回到 handle_int 中来。我们识别了时钟中断，然后进入 timer_irq 分支进行处理，分支跳转了 schedule 函数，这就是用来进行进程调度的函数。

它有一个参数 yield，表示是否强制让进程退出。

首先，schedule 函数有一个静态变量 count，表示当前进程剩余的时间，一个 PCB curenv 表示当前进程。

接着我们考虑所有需要切换进程的情况：

  1. yield == 1，强制退出当前进程，换另一个。

  2. count <= 0，表示时间片结束。

  3. e == NULL，这是刚内核刚刚完成初始化，第一次产生时钟中断的情况，需要调度一个进程。

  4. env_status != ENV_RUNNABLE，进程阻塞或完成，可以换另一个。

我们先考虑进程因为时间片结束而切换的情况：先从被调度的队列中移出来。然后判断：这个进程是阻塞了或完成了吗？还是只是因为时间结束而挂起？如果它还能继续运行，我们得把它移动到链表末端，让它等待下一次的调度。这一步一定要做，否则下次从链表中取出 PCB 的时候会重复取到这一个时间结束的 PCB。

Lab3 Exam

Lab3 Exam 基本上沿袭之前的出题模式，考察进程调度算法的实现。我们只需要按照题目意思一步一步实现就可以了。

如果你像我一样只是为了完成上机测试，那么不需要完全了解题目中的调度算法的原理，最重要的是看明白这个调度算法的行为。

我们要实现一个名为抢占式短作业优先Shortest Remaining Time First, SRTF的算法。核心内涵是：

  1. 维护一个独立的调度队列，与默认的时间片轮转算法队列互不干扰。

  2. 当时钟中断产生时，若 SRTF 调度队列中存在尚未运行完所需时间片的进程，则选取其中剩余运行时间最少的进程调度。如果多个进程的剩余运行时间相同，则选择 env_id 最小的进程调度。

  3. 从 SRTF 调度队列选取进程后，你仅需使其运行一个时间片，并在下个时钟中断产生时根据第二条规则，重新选择进程调度。

  4. 如果 SRTF 调度队列为空，或其中的所有进程均已运行完所需的时间片，则使用课下实现的 RR 算法调度 RR 调度队列中的进程。

  5. 为了保证 RR 调度的连续性，当 SRTF 抢占结束后，RR 算法应该从上次被抢占的地方继续运行。

下一步要修改 kern/sched.c 中的调度函数 schedule，增加 SRTF 的算法。一个可能的实现思路如下：

  1. 遍历 SRTF 调度队列，如果存在剩余时间不为空的进程，说明有尚未完成运行的进程，进入 SRTF 算法。如果不存在这样的进程，或者 SRTF 调度队列为空，则使用 RR 算法进行调度。

  1. 如果有尚未完成运行的 SRTF 进程，则遍历整个 SRTF 队列，选取剩余时间片最小的进程。如果有多个剩余时间片相同的进程，选 env_id 最小的进程。如果剩余时间片为 0，移除之。

  1. 如果找到了这样的进程，那么将剩余时间片减 1，并调用 env_run 调度该进程。

Lab3 Extra 和 CO P7 有点关系，但如果没有做到 P7 也并不影响。

Extra 的任务是完成 9 号异常的异常分发和处理。我们根据传入的参数 code 来实现三种不同的异常处理逻辑。

9 号异常由 break 指令触发。

格式：break code，code 在 32 位机器码中占据 25 至 6 位。

  1. 若 code == 0x0，表示用户断点，直接退出异常处理，继续执行后续指令。

  2. 若 code == 0x6，数组访问越界触发，程序会将数组长度存入a0（4号寄存器），要求对访存指令（当前break指令的下一条指令）的立即数(‘imm‘)部分进行检查和修改，对应索引为(‘imm>>2‘)：a0（4号寄存器），要求对访存指令（当前break指令的下一条指令）的立即数(`imm`)部分进行检查和修改，对应索引为(`imm >> 2`)：a0（4号寄存器），要求对访存指令（当前break指令的下一条指令）的立即数(‘imm‘)部分进行检查和修改，对应索引为(‘imm>>2‘)： - 若越界的索引为负数，则将立即数设置为 0。

  • 若越界的索引大于等于数组长度，则修改立即数，使得索引为数组长度减1，即数组的最后一个元素。

  1. 若 code == 0x7，除零异常触发，要求获取将要发生错误的除法指令（当前 break 指令的下一条指令），并且修改除数为 2。

一些指令的机器码如下所示：

指令 31 - 26 25 - 21 20 - 16 15 - 11 10 - 6 5 - 0
div 000000 rs（被除数） rt（除数） 00000 00000 011010
指令 31 - 26 25 - 21 20 - 16 15 - 0
lw 100011 base rt imm
sw 101011 base rt imm

另外不同的异常处理结束后还需要输出处理的结果。

我们首先按题目要求更改异常向量组、写好函数定义，此处不再赘述。

Thinking 3.6

开启时钟中断

时钟中断由 handle_int 函数处理。当识别到时钟中断时，它跳转 timer_irq 分支，并执行 schedule 函数。该函数调度下一个 PCB 并执行 env_run 函数启动进程。env_run 调用 env_pop_tf 还原上下文的时候，重置了时间片，并回到用户态，开启中断。操作系统可以继续响应时钟中断了。

关闭时钟中断

在 kern/entry.S 中，异常分发函数通过 and t0, t0, ~(STATUS_UM | STATUS_EXL | STATUS_IE) 关闭响应时钟中断，然后跳转到相应的异常处理函数。

Thinking 3.7

  1. 时钟中断产生

  2. 进入异常分发程序

  3. 查找异常向量组，进入异常处理函数

  4. 进入进程调度函数 schedule

  5. 在满足下述条件时切换进程：

  6. yield == 1，强制退出当前进程，换另一个。

  7. count <= 0，表示时间片结束。

  8. e == NULL，这是刚内核刚刚完成初始化，第一次产生时钟中断的情况，需要调度一个进程。

  9. env_status != ENV_RUNNABLE，进程阻塞或完成，可以换另一个。

  10. 调用 env_run 执行新进程
