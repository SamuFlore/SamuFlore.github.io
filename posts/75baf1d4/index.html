[BUAA OS Lab5]文件系统

从微内核设计说起

一般来说，操作系统访问磁盘的流程是这样的：

  1. 用户进程发起系统调用，访问磁盘

  2. 磁盘返回一些信息给用户进程

  3. 用户进程对文件进行读写

  4. 文件重新写回磁盘

但是 MOS 是具有微内核的特点，即内核的功能是极大程度地精简过的。对于 MOS 来说，我们还需要引入文件服务进程，即 FS。

  1. 用户进程向文件服务进程发送一个请求

  2. 文件服务进程负责从磁盘中找到文件，给到用户进程（换入）

  3. 用户进程对文件进行读写

  4. 读写结束后，用户进程向文件服务进程发送一个请求

  5. 文件服务进程将文件放回磁盘（换出）

我们首先来看看文件服务进程如何与磁盘进行交互。

磁盘操作

磁盘的数据结构

在正式开始之前，我们先看看文件在磁盘上是如何组织的。

在我们的实验中，我们可用的磁盘数目是 1 块，大小是 4 MB。磁盘中的基本单位是磁盘块，一个磁盘中有 1024 个磁盘块，因此每个磁盘块的大小是 4 KB——刚好等于一个页的大小。

这 1024 个块的功能都一样吗？不，第 0 个和第 1 个块有其特殊的功能。

  • Block 0：引导扇区和分区表Boot。

  • Block 1：超级块Super Block。超级块存储文件系统的基本信息，比如魔数Magic Number，磁盘大小以及根目录的位置。

NDIRECT 大小为 10。这两个成员是什么意思？正如之前所说，FCB 不直接存储文件内容，但它必须给我们提供文件内容的地址，这就要用到 f_direct 和 f_indirect 两个成员了。

f_direct 表示 FCB 直接存储的磁盘块索引。NDIRECT 为 10 表示一个 FCB 至多可以直接存储 10 个磁盘块的索引。倘若我们要通过 FCB 获取文件内容，就需要找到索引，然后取读取对应的磁盘块。

虚拟地址空间布局

FS 进程的地址空间（kuseg 段）是这样使用的：当我们需要使用磁盘块的数据时，我们将磁盘块的数据复制到 FS 进程的虚拟地址空间的某处，这就叫「换入」。为了简化设计，每个磁盘块会被换入到虚拟地址空间的一个固定的位置。

基地址是 0x10000000，我们记为 base。最高地址是 0x50000000，一共 1 GB 的空间，磁盘块被换入后就保存在这地址范围内。

具体来说，FS 进程发送请求，要换入第 n 个磁盘块。磁盘接受请求后，将会把磁盘块的内容装入从 base + n * 4 KB 开始的 4 KB 空间内。

之后，用户进程会和 FS 进程通信，获取这些数据。

我们可能会对数据进行修改，也有可能只是读取而不改任何内容。如果我们改了，就必须通知 FS 进程更新它虚拟地址空间的数据，然后让 FS 进程将新的数据写回磁盘，这就是「换出」。如果只是读取，那么就没必要让 FS 进程写回，这样可以节约时间成本。当然，硬让 FS 进程写回也不是不可以，只是没必要。

我们将利用「脏位」来实现这一功能，这在后面会提到。

空闲块管理与位图操作

接着我们有检查磁盘块是否为空闲的函数。通过读取位图块来得知该磁盘块是否为 FREE。

在此之前，我们必须先将超级块和位图块读入内存，否则我们无法直接查询。

超级块被读入 *super 中，而位图块用一个 uint32_t 类型的数组来承接。前面说到我们只需要 128 字节来表示位图块，这也就是说我们只需要 128 / 4 == 32 个 uint32_t 元素来表示。

首先，如果 super == 0，说明我们还没有将超级块读入内存，更无从谈起其它磁盘块的空闲与否，所以直接返回不空闲。还有一种情况是 blockno 超出了 1024，也返回不空闲。

接下来我们正式来查位图。我们先从磁盘块编号得知它在第几个 uint32_t 元素中：每个元素占 32 bits，blockno / 32 得到答案。

接着，每个元素内又有 32 bits，我们要知道究竟是哪一位代表这个磁盘块。用取余数的方法构建一个常数，按位与得到最终的答案。

数据结构介绍

USER 进程发来的请求会被包装成 struct Open 结构体：

首先是一个 struct File * 类型的指针，指向要打开文件的 FCB。

o_fileid 是一个 0 到 1023 的数。在 FS 进程中，一共有 1024 个 struct Open 结构体供我们使用：

也就是说每次 USER 进程发送打开文件的请求时，FS 进程需要在这 1024 个结构体中分配一个未使用的结构体来处理这个请求。

o_mode 表示文件打开的模式。

struct Filefd 是什么结构体呢？在 FS 虚拟地址空间的 0x60000000 开始，有一个 4 MB 的空间，内含 1024 个页。每个页中有一个 struct Filefd 结构体，同样描述了一些文件的信息。特别的是，这 4 MB 地址空间将与 USER 结构体进行共享，并且每个 struct Filefd 结构体和一个 struct Open 结构体 一一对应。

一共五个成员变量，分别是：

  • fd_dev_id：设备种类，目前是一个恒定字符 f。设备种类包括文件系统、管道和控制台。

  • fd_offset：一个「指针」，表示当前读取到文件内容的第 fd_offset 字节。这有点类似于 C 中的文件操作。

  • fd_omode：文件打开的模式。等于 struct Open 中的 o_mode。

  • f_fileid：等于 struct Open 中的 o_fileid。

  • f_file：文件 FCB 的 拷贝。这不是一个指针！

总的来说，struct Filefd 和 struct Open 像一对孪生兄弟。struct Fd 像一个父类，而 struct Filefd 像一个继承父类的子类。

既然有这样一层继承关系，实际上我们可以通过指针的强制类型转换来实现 struct Fd * 和 struct Filefd * 之间的互换。但注意，只有 fd_dev_id == 'f' 时，才说明这块内存存储的是 struct Filefd 类型。否则我们不能轻易地将 struct Fd * 转为 struct Filefd * 类型，然后访问最后两个字段。

file.c

我们先来看 file.c 中提供的 API。

open 函数一开始，我们调用 fd_alloc 分配一个 struct Fd，这是什么意思？

之前我们说，在 FS 进程的虚拟地址空间，0x60000000 以上有 1024 个 struct Filefd，每个结构体都在一个单独的 4 KB 页面，并且这个页面是 FS 进程和 USER 进程共享的。这也就是说 va1(USER) 和 va2(FS) 映射到同一个物理页面。

但是在 FS 进程中我们有 1024 个 va2(FS)，在 USER 进程中我们却只有 32 个 va1(USER)！这也就是说它们并不是一一对应的关系。因此我们需要遍历这 32 个位置，看看哪里空出来，然后与 FS 进程中的地址映射到同一个物理页。

这 32 个 struct Fd 在 USER 虚拟地址空间的 0x60000000 低 4 MB 处。

分配之后，我们调用 fsipc_open，将 fd 作为 dstva(USER)。在 FS 进程侧，它会传回一个页面，在 USER 进程侧，用 fd 来接住它。

下一步我们做的事情是这样的：我们在 USER 进程的 0x60000000 高处开辟 32 * 4 MB 空间，并划分成 32 份。对于刚才分配的 fd，从 0 开始计数是第 i 块，就把文件内容映射到 32 * 4 MB 空间的第 i 个 4 MB 空间中。

va2data 函数是一个辅助函数，功能是给定 fd，根据刚才说的方法换算出 0x60000000 上方的虚拟地址。

然后我们要将 fd 强制转换成 struct Filefd * 类型。毕竟它在 FS 进程地址空间就是 struct Filefd，我们只是用 struct Fd 来承接它。如果要用独属于 struct Filefd 的成员，我们还是得进行转换。

下一步，我们遍历整个文件大小，每次偏移 PTMAP == 4096，使用 fsipc_map 将文件内容映射到 USER 虚拟地址空间。fsipc_map 的作用是将 fileid 对应文件的第 i 字节所在的磁盘块内容映射到虚拟地址 va + i(USER) 处。

最后返回给用户程序的是 fd 在 0 到 31 中的序号。

fd.c

在 fd.c 中也有值得注意的函数。

首先是 read：

read 接受三个参数：fdnum 表示文件对应的 struct Fd 序号，buf 表示将文件内容读到 buf 中，n 表示长度。

调用 dev_lookup 本质上是根据 fd_dev_id 来查询当前设备的种类，对于 Lab 5 而言，一定返回的是 struct Dev devfile，即文件系统。

后四个成员是四个不同的函数。

第二步获取打开方式，我们之前已经提到了必须用 O_ACCMODE 掩码。

第三步我们调用的是 struct Dev 的 dev_read 函数，实际上执行的是 file_read 函数。

这个函数很简单，参数比 read 多一个 offset，传入的是 fd -> fd_offset，即文件的「指针」。如果 offset 已经超出了文件大小，返回 0；如果 offset + n 超出文件大小，则截断，只读到文件末尾；最后用 memcpy 将数据写入 buf。最后返回读入的字符数 n。

最后，我们修改「指针」的偏移量。

下一个关键的函数是 write：

write 的作用是将 buf 的内容写入 fdnum 对应的 0x60000000 上方的对应 4 MB 区域。它调用的 dev_write 本质上是 file_write。file_write 会动态调整文件大小：如果 offset 加上要写入的长度 n 大于当前文件长度，就调用 ftruncate 函数将长度扩大，但不能超过最大长度。在写入内容后，调用 fsipc_dirty 将该页面标记为脏页。这里涉及到的函数较多，我们只需了解其功能即可。

在 file.c 和 fd.c 中还有一些 API 受限于篇幅无法完全讲到，但我们可以通过已经讲到的这些清楚的看到用户进程、文件服务进程以及磁盘之间的关系。

IDE 磁盘驱动

关于用户进程-文件服务进程-磁盘的交互逻辑，我们还差最后一块拼图：从磁盘读取数据和将数据写回磁盘，分别使用 ide_read 和 ide_write 函数。

但请放心，我们并不会用上真正的、物理意义上的磁盘，而是一个磁盘镜像。

IDE 磁盘操作

关于磁盘的结构，包括柱面、磁道、磁头、扇区等内容，请回顾计算机组成课程内容。

我们可以采用「柱面-磁头-扇区」的方式寻址，即CylinderC-HeadH-SectorS寻址方式。但在我们的实验中，我们将磁盘简化成了一个一维的结构，只需要通过扇区号就能访问对应的扇区。

每个扇区的大小是 512 Bytes。因此每个磁盘块大小为 8 个扇区。

注意

磁盘块是逻辑单位，在磁盘中寻址、读写只以扇区为单位而不以磁盘块为单位。


现在我们来看看 ide_read 和 ide_write：

参数的含义如下：

  • diskno：磁盘号。这不是 0 到 1023 的磁盘块号，而是磁盘的编号。我们至多可以挂载 2 块磁盘，但实际只用到了 1 块，因此一般传入 0。

  • secno：扇区号。表示读取内容的起始位置。

  • dst：所读内容保存的虚拟地址（FS 进程）。

  • nsecs：扇区数。接下来读取 nsecs 扇区的内容。

我们在检查参数有效性之后，遍历从 secno 到 secno + nsecs 的所有扇区，准备读入数据。如何读入呢？这就需要用到一种名为内存映射 I/OMMIO 的技术了。

MMIO

我们访问范围属于 0x180001F0 到 0x180001F7 的物理地址，可以得到 IDE 磁盘的一些数据。

然而，我们不能之间访问物理地址。所以我们选择将物理地址转换为 kseg1 段的虚拟地址，即加上 0xA0000000。

为什么？

kseg1 段的访存不经过 Cache。假设我们用 kseg0 段（经过 Cache）虚拟地址：第一次访问地址，得到一个数据，然后因为某些原因，磁盘中该数据发生了更改。当再次访问同一个地址时，我们将从 Cache 中得到缓存的旧数据而不是发生更改后的新数据，这就会导致错误。

那么这个范围内的虚拟地址究竟保存的是什么呢？我们来看这样一张图：

别的其实都没什么问题，但你看到偏移为 0x0 的时候可能甚是费解：为什么能读出 4 Bytes？

这是因为这一段虚拟地址对应的不是内存中的空间，它只是一个代号，我们通过它访问的不是内存，而磁盘中的寄存器！访问 0x180001F0 + 0xA0000000 时，我们本质上是在访问一个 32 bits 的寄存器。访问其它地址时，本质上是在访问一个 8 bits 的寄存器。

访问每一个寄存器时都有不同的作用，比如访问 32 bits 寄存器时，我们得到的是这一次读或写的内容。具体功能请看下表：

我们至多可以写入 28 位扇区号，也就是说我们支持访存 228 个扇区。

这就是所谓的内存映射 I/O。

现在我们回到 ide_read 来看看如何利用这些寄存器实现数据读入。

wait_ide_ready 循环检查 MALTA_IDE_STATUS，也就是偏移为 0x7 的寄存器——设备状态，直到磁盘准备就绪。

接下来我们填入扇区信息，包括扇区号，扇区数等等。这里需要用到 syscall_write_dev 这个系统调用。它的作用是将给定的数据写入 kseg1 段的虚拟地址（也就是 MMIO 的这些特殊虚拟地址）。

每一步，都往一个寄存器中写入相关的内容。在写入偏移量 0x6 的寄存器时，通过位运算将三种数据融合到一起。

第六步主动改变磁盘状态，提示参数填写完毕，准备开始读取。接着等待磁盘准备就绪。

下一步，由于 0x0 偏移的寄存器大小只有 4 Bytes，这就是说我们每次只能读 4 Bytes 内容。所以我们用一个 for 循环遍历一个扇区，步长 4 Bytes，将内容读到虚拟地址处。

循环变量 offset 从 0 开始，每读完一个扇区，offset 都会加上扇区的大小，i 是扇区内部的偏移量，因此虚拟地址应该是 dst + offset + i * 4。

这样，我们就实现了从磁盘往内存中读入数据。


ide_write 与 ide_read 几乎一模一样，如果你从某处跳转而来，请你阅读上方 ide_read 的逻辑分析。

Thinking 5.1

设备寄存器的数据变化可能会通过硬件发生。这种变化不会经过 Cache。当硬件改变寄存器的值的时候，寄存器和 Cache 的值会出现差异。此外，当系统断电或崩溃，Cache 会被清空，导致数据不一致或丢失。
