[BUAA OS Lab2]内存管理

序

在工作能够完成的时限内，工作量会一直增加，直到所有可用时间都被填充为止。
——《经济学人》，1995

这是西里尔Cyril·诺斯古德Northcote·帕金森Parkinson提出的一句话。在 1958 年，帕金森把这句话扩充成一本书，名为《帕金森定律：对于进度的追求》，同时凝练出了著名的帕金森定律：

在预算之内，支出的需求会一直增加，直到所有资源被用完为止。

在计算机领域，这一定律同样适用。给 128 MB 可用内存，软件就占用 128 MB；给 16 GB 可用内存，软件就占用 16 GB。内存越来越大，但软件用掉的也越来越多，却仿佛没有本质上的飞跃。所以，只要帕金森定律仍然生效，无论我们有多大的内存，内存都是稀缺的资源，做好内存管理是每个开发者的本职工作。

page_init

说起来，我们为什么要用链表呢？

是这样的，面对这么多物理页面，我们自然而然有一个需求：维护所有的空闲页面，如果有需要，就分配出去。我们现在手上有的是 pages 这个指针，增删空闲页面的时候只能去遍历，时间复杂度 O(n)，太没有效率了。所以我们再维护一个链表，将所有的空闲页面串起来，这样时间复杂度优化到 O(1) 了。

接下来这一步值得仔细审视一下：

我们先回顾一下现在的内存区域到底是什么样子的：

我们有 npage 个物理页面，就需要 npage 个 Page 结构体来管理，但是这些结构体也是需要占用物理空间的，这就形成了一种自映射：pages 数组所在的物理页面，会映射到 pages 数组中的某个 Page 结构体。所以说，直到这一步，我们已经用掉的物理空间不仅仅是 .text, .data 和 .bss，还有 pages 自己本身。幸好，我们掌握了 pages 的末尾在什么位置——freemem。

freemem 之前的虚拟空间我们都用掉了。换句话说，freemem 映射到的物理页之前的物理页我们都用掉了。我们需要将这些用掉的物理页重新映射回对应的 Page 结构体，将它们的 pp_ref 标记为 1。

这听起来很绕，但实现起来并不复杂。我们有 freemem 的虚拟地址，先用 PADDR 宏将其变为物理地址，再用 PPN 得到它是第几个物理页。接下来就是很妙的一步：因为 pages 各元素和物理页一一对应，第 i 个 Page 结构体就对应第 i 个物理页，我们直接从 0 遍历到 PPN(PADDR(freemem)) 就能得到所有用掉的物理页了！

最后，再将没有用过的物理页的 pp_ref 置为 0。并将这些页面加入到 page_free_list 链表中，通过它就能查到所有空闲页面。

提示

这也就是说新加入的空闲页会在链表靠前的位置。

首先我们有页目录的基地址 *pgdir，以及虚拟地址，用 PDX 宏获取页目录偏移量，二者相加得到页目录项的地址。PDX 就是取出虚拟地址的 31 到 22 位。

提示

pgdir_entryp 解引用后得到的是一个页目录项，明明由高 20 位的物理页号和低 12 位的权限位组成，为什么可以直接把权限位和物理地址拼在一起得到？

根据我们之前提到的公式（如果你不记得，请回顾页控制块），PA = PPN << 12，保证了物理地址的低 12 位恒为 0。因此我们直接往低 12 位写入权限位，而高 20 位自然是物理页号。

如果 create == 0，我们就不会创建一个新的页表项，而是直接让 va 对应的页表项地址为 NULL，并直接 return。

tlb_out 是一个汇编函数：

Lab2 Extra

Lab2 Extra 涉及到页表自映射的原理和实现。

注意

左移 << 12 位等价于乘以页面大小 4KB，左移 << 2 位等价于乘以表项大小 4byte。之后不会再强调这一点了（）

在正式开始之前，我们需要了解页表自映射是怎么一回事。

一直以来，我们都只能通过 kseg0 段的虚拟地址来访问页目录和页表，这限制了我们只能在内核态访问它们。现在我们希望在用户态可以通过只读的方式访问到页表。

换句话说，我们需要从 kuseg 段中选一部分虚拟地址出来，将它们映射到页表所在的物理页面。虚拟地址空间很大，不用白不用（当然这是相对于物理地址空间而言的）。

最简单的办法是这样：我们假设从 base 到 base + 4MB 这个前闭后开的 4MB 区域映射到 1024 张页表，每张 4KB。然后我们还需要一个 4KB 空间，分为 1024 * 4byte，作为一级页表（页目录），每个页目录项（4byte）存储一个页表的基地址。

可以证明这样一件事：base 是 4MB 的倍数时，在 [base, base + 4MB) 这个区间内，地址的高 10 位是保持不变的。

这有什么用呢？这意味着当我们拿到虚拟地址开始查页目录的时候，我们找到的永远是同一个页目录项：pgdir + PDX(va) << 2，然后我们就会根据这个页目录项找到同一个页表，再去查页表项，最后得到物理页。

既然这个页目录和页表总是绑定在一起，那我们不如把它们合二为一。

具体是这样做：既然我们查到的永远是同一个页目录项，并且这个页目录项会指向同一个页表，而我们现在又要将二者变成同一个东西，那我们干脆让这个页目录项指向这个页目录本身就好了。这样的话，页目录同时又成了页表，实现了合二为一。

我们的内存布局就会变成这样：

那凭什么仅仅只让页目录的某一个特殊项指向页目录自己的物理地址，就能让所有页表自动地平铺到 [base, base + 4MB) 这个虚拟地址空间呢？
对于 [base, base + 4MB) 这一段空间，我们知道 base + i << 12 可以得到第 i 个页表的基地址。当页目录的某个特殊项指向自己的物理地址，页目录本身也成了一个页表，我们假设 base + k << 12 可以找到页目录的基地址，它同时是第 k 个页表的基地址。

现在对于虚拟地址空间内的任何一个虚拟地址 va，我们按照虚拟地址翻译到物理地址的过程走一遍：

  1. 取一级页表偏移 PDX(va)，找到页目录项：pgdir + PDX(va) << 2。按之前假设的页目录基地址，这应该是 base + k << 12 + PDX(va) << 2。由于第 i 个页目录项对应的是第 i 个页表的基地址 base + i << 12，我们这里找到的事实上是第 PDX(va) 个页表的基地址 base + PDX(va) << 12。

  2. 取二级页表偏移 PTX(va)，得到页表项：base + PDX(va) << 12 + PTX(va) << 2。

啊哈！这真是太神奇了。相当于我们不用苦哈哈地去管什么地址翻译流程，我们只需要从 base 开始偏移两次：一次是高 10 位 PDX(va)，一次是中间 10 位 PTX(va)，就能找到虚拟地址对应的页表项，得到物理页框号了！

有了对自映射的基本认知，我们来看看问题是什么。

我们需要完成四个函数，模拟自映射的建立、查询和解除：

  • create_self_map(Pde *pgdir, u_int asid, u_long base)

    • 首先遍历 [base, base + 1024 * PAGE_SIZE) 的连续 4MB 虚拟地址空间，检查其是否与物理页进行了映射。如果有，则清除，并将计数器 count 加 1。
    • 然后将所有页表自映射到 [base, base + 1024 * PAGE_SIZE) 的地址空间。页目录项必须有效且不可写。
    • 返回 count 的值。

  • pte_va(u_long base, u_int pdeno, u_int pteno)

    • 给定自映射虚拟地址空间的首地址 base ，计算使用自映射访问第 pdeno 张页表的第 pteno 个页表项时，需要使用的虚拟地址。

  • pde_va(u_long base, u_int pdeno)

    • 给定自映射虚拟地址空间的首地址 base ，计算使用自映射访问页目录中的第 pdeno 个页目录项时，需要使用的虚拟地址。

  • remove_all_self_map(Pde *pgdir, u_int asid)

    • 首先遍历所有页目录项，找到其中用于实现自映射的页目录项，将其清空来解除自映射，并用计数器 count 记录解除自映射的数量。
    • 通过上述页目录项的地址相对页目录首地址的偏移，算出自映射虚拟地址空间的首地址 base。
    • 遍历 [base, base + 1024 * PAGE_SIZE) 的虚拟地址空间，清楚 TLB 缓存。
    • 返回 count 的值。

Thinking 2.4

  • ASID 的必要性

    1. 每个进程都认为自己拥有完整的虚拟内存空间，因此它们所用的虚拟地址往往会产生冲突。当切换进程时，如果 TLB 中残留的原来进程的表项，就可能导致访问的数据错误。使用 ASID 进行区分可以隔离不同进程的虚拟地址。
    2. 切换进程时，可以只更改 EntryHi 寄存器，而不用刷新整个 TLB。下次切换回来时，TLB 中还能保存一部分表项，加快访问速度。

  • 最大数量：28 = 256 个地址空间。

  1. tlb_invalidate 调用 tlb_out。

  2. tlb_invalidate 删除指定的 TLB 旧表项。

  3. tlb_out 的解释如下：函数首先将 Entry_Hi 寄存器的值保存在 $t0 中，然后将 $a0（指定的 Key）写入。

然后根据这个 Key 去找旧表项，索引会存入 Index。

接下来取出这个索引，并和 0 比较：如果小于 0，则表示这个表项不存在，跳转 NO_SUCH_ENTRY 函数。

如果大于等于 0，则表项存在，往 Entry_Hi, Entry_Lo0 和 Entry_Lo1 中写入 0。
