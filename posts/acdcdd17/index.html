[OS Lab1]内核、启动和 Printf

什么是内核？

好吧，在正式开始之前，我需要问自己一个问题——“什么是内核？”

操作系统内核是属于操作系统的，它是操作系统的一个组成部分。

操作系统内核拥有操作所有硬件的权限，但不会让用户直接操作——一般情况下，用户使用操作系统的时候都是在用户态，用户态没有能力操作这么多。

有的时候会陷入内核，有的时候不会。

启动操作系统

按下电脑电源之后，是不需要我们自己去启动操作系统的；重启电脑之后，原先的操作系统还是能保留在电脑里的。我们的生活经验其实说明了两件事：

  1. 通电之后，有什么东西让内核运行起来了，或者内核自动运行起来了。

  2. 内核不在 RAM 里，那个东西也不在（如果有的话）。

其实，再仔细考虑一下，就知道这个“东西”是真实存在的。我们假设这个“东西”不存在。

如果内核不在 RAM，那还能在哪里？这个问题只能从非易失性存储器中找答案。

如果内核在磁盘，可以吗？答案是不行。一个程序要能运行，与它相关的指令和数据必须先加载到内存里，让 CPU 直接访问，程序才能运行起来。而这恰好是内核该做的，这就发生矛盾了——内核想要运行起来，需要被自己加载到内存，但是内核需要运行起来才能把自己加载到内存。

✋😭🤚一定不要自己加载自己✍️✍️✍️

事实上，这正是操作系统设计的时候遇到的一个知名难题：硬件是在软件的控制下执行的，而当刚上电的时候，存储设备上的软件又需要由硬件载入到内存中去执行。可是没有软件的控制，谁来指挥硬件去载入软件？

这就叫做 pull oneself up by one's bootstraps。操作系统的启动就叫 boot。

回到刚才的话题，如果内核在 ROM 或 FLASH，可以吗？答案是可以，但是不好。理由如下：

  1. 学过计组的朋友们都知道，CPU 的可寻址空间是有限的。ROM 或 FLASH 确实可以被 CPU 直接访问，但它们的空间都挺小的，肯定装不下现代操作系统的内核。

  2. ROM 是写死的，这意味着内核放在其中就无法更改，不利于移植或更新。

  3. 如果 CPU 通电后，操作系统直接启动，意味着一台计算机只能启动一个操作系统。

所以设计人员从这句充满哲理的谚语中找到了一个解决办法，那就是分两个阶段启动操作系统，并设计一个叫 bootloader 的程序做先导。

通电

CPU 通电，取指寄存器复位。

Stage1

相关硬件初始化、初始化 RAM、将 Stage2 代码复制到 RAM，跳转 Stage2 入口。

Stage2

相关硬件初始化、载入内核镜像到 RAM、为内核设置启动参数，跳转内核入口。

完成

操作系统开始运行。

有 bootloader，就可以支持运行更大的操作系统，也可以支持选择多系统，解决了这个难题。

等等

事实上，以上介绍的是真实操作系统的大致启动流程。在这门课中，我们使用的是 QEMU 模拟器模拟一台没有操作系统的裸机环境，这个启动过程被极大地简化了。QEMU 支持直接加载 ELF 格式的内核，相当于提供了一个 bootloader。因而启动过程已经简化成了加载内核到内存，随后跳转到内核入口，启动就完成了！

MOS 内核文件组织

内核文件大概是由这些目录及其下的文件构成的：

  • init：内核初始化相关代码

  • include：存放系统头文件

  • lib：库函数目录

  • kern：存放内核的主体代码

  • tests：存放测试程序

  • tools：存放实用工具，比如 readelf

  • target：存放编译的产物（执行 make 后产生）

还有在根目录的 Makefile，这是整个内核文件组织的地图。

什么是 ELF ？

刚才若干个地方都提到了 ELF，那什么是 ELF？
ExecutableE and LinkableL FormatF 是 Unix 上一种常用的目标文件格式。我们都知道，用 gcc -c 可以得到目标文件 *.o，然而由于整个程序的布局还没有确定，这些目标文件中的跳转指令等等的目标地址还没有确定下来。在链接阶段，所有目标文件会被链接在一起，过程中链接器会填写这些空缺的地址信息[1]。

其实，不仅是目标文件，可执行文件也是 ELF 格式。

执行 Makefile 得到的 ./target/mos 也是一个 ELF 文件。我们需要了解一下 ELF 文件到底包含了什么内容。

ELF 文件头的前 16 个字节是魔数Magic Number，这 16 个字节的前 4 个分别是 0x7F, E, L 和 F（后三个字母是其对应的 ASCII），用来说明这是一个 ELF 文件。

然后是关于段头的一些信息。首先我们想一想，如果要通过 ELF 头给的信息去访问所有的段头表，我们需要知道什么信息呢？我们首先要知道第一个段头表在哪，意思就是知道第一个段头表相对于 ELF 头的偏移量是多少，也就是 e_phoff；然后，段头表是紧密排列的，我们只要每次将指针移动一个段头表那么多距离，就能访问到下一个段头表了，意思就是我们还得知道每个段头表的大小，也就是 e_phentsize；但是，我们也不能无限地往后找，不然就超出内存范围了！所以我们还要知道一共有多少段头表，即 e_phnum。

对于节头来说，同样有这三个信息，分别是 e_shoff, e_shentsize 和 e_shnum，不再赘述。

e_ehsize 是 ELF 头自身的大小。

包含程序中各个段Segment的信息，段的信息在程序运行时使用。

这一个段头有 8 个字段，每个 4 字节，所以一个段头是 32 字节。

但我们重点关注 p_type 等于 1 的 PT_LOAD 类型。PT_LOAD 类型表示这个段需要加载到内存中，而其它全部类型都不需要加载。

p_offset 事实上是基于 ELF 头的偏移。它指向了这个段头对应的段的位置。

接下来 p_vaddr 和 p_paddr 表示装载时要用到的虚拟地址和物理地址。

然后是 p_filesz 和 p_memsz。第一个 p_filesz 表示该段数据在文件中的大小，后面 p_memsz 表示该段数据在内存中应当占的大小。举个例子：考虑一个 C 程序，它有一个全局变量，但是没有初始化。在加载这个程序到内存的时候，我们肯定需要为这些未初始化的东西分配空间，但是又不需要填充任何值。所以说 p_memsz 永远大于等于 p_filesz（如果没有未初始化的全局变量，那自然二者是相等的）。

事实上，当 p_memsz 大于 p_filesz 时，会首先将文件中记录的数据加载到对应的 VirtAddr 处，然后向内存中填充 0 直到二者相等，这也就解释了为什么未初始化的全局变量会自动初始化为 0。

节头表

包含程序各个节的信息，节的信息在程序编译和链接的时候使用。

这一个节头有 10 个字段，每个字段都是 4 字节，所以一个节头是 40 字节。

sh_offset 事实上是基于 ELF 头的偏移。它指向了这个节头对应的节的位置。

sh_size 告诉我们节的大小。

  • 段头表的表项和节头表的表项可以重复，它们只是数据的两种视图。

使用 readelf 工具查看一个 ELF 文件，可以看到各个段的信息，这些信息包括了该段的数据相对于 ELF 文件头的偏移量、该段最终需要被加载到内存的哪个位置等等。也就是说 QEMU 模拟器通过读取我们编译出来的内核文件，找到正确的位置，将各种数据加载到相应的位置，并将 CPU 的控制权交给内核。

MIPS 内存布局

那么什么位置才是正确的位置？

依旧学过计组的朋友们都知道，用户编写程序时使用的地址称为虚地址逻辑地址，对应的是虚存空间虚拟地址空间，这些地址会通过一个叫做 MemoryM ManagementM UnitU的硬件模块映射到物理地址[2]。

在这门课的 MIPS 体系结构中，虚存空间大小为 4 GB，并规定了这 4 GB 如何划分：

MIPS 内存布局

  • kuseg 0x00000000 - 0x7FFFFFFF, 2 GB：这是用户态下唯一可用的地址空间，要通过 TLB 进行虚地址到物理地址的变换，存取都经过 Cache。

  • kseg0 0x80000000 - 0x9FFFFFFF, 512 MB：这是内核态下可用的地址，MMU 将虚地址最高位清零（& 0x7FFFFFFF）来得到物理地址，存取都经过 Cache。

  • kseg1 0xA0000000 - 0xBFFFFFFF, 512 MB：这是内核态下可用的地址，MMU 将虚地址高三位清零（& 0x1FFFFFFF）来得到物理地址，存取不经过 Cache，一般用于访问外设。

  • kseg2 0xC0000000 - 0xFFFFFFFF, 1 GB：这是内核态下可用的地址，需要 MMU 中 TLB 将虚地址转换为物理地址，存取都经过 Cache。

要把内核放在哪一段呢？TLB 的运作是需要操作系统进行管理的，因而不能把内核放在 kuseg 或 kseg2，只能在 kseg0 和 kseg1 中选。但 kseg1 不经过 Cache，一般来说用于访问外设，所以最后我们决定把内核放在 kseg0 中。

难点分析

Lab 1 的难点有如下几个方面：

  • 理解 ELF 文件结构：首先要理解 ELF 头、段头表、节头表的关系，并了解如何通过 ELF 头访问各个段头表和节头表。

  • 了解从内核如何从源代码到可执行文件：需要看懂 Makefile，了解各指令在做什么事，知道如何指定 .text 等的位置，知道如何写 Linker Script 等等。
