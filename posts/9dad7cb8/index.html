控制台Console

在 Lab 5 的博客中，我们提到一共有三种可能的「设备」。包括「文件系统」、「管道」和「控制台」。

并且提到在 user/lib/fd.c 中涉及到的函数一般来说是三个设备通用的，而在 fs/fs.c 中涉及到的是文件系统专用的函数。在 fd.c 中的函数，会调用 dev_lookup 函数获取不同设备对应的函数，进行分发处理。

在具体介绍「管道」和「Shell」之前，我们先来介绍一下「控制台」。

什么是「控制台」？

首先，我们明确一个概念：「管道」、「文件系统」和「控制台」都不是可执行文件，也不是一个进程。文件服务进程是一个进程，但是「文件系统」不是一个进程。准确地来说，它们只是一种数据结构。

控制台是用来处理外设输入，并输出到屏幕的一种设备。在我们的 MOS 中，它主要处理键盘输入。每个进程都可以调用控制台，只需要调用 user/lib/console.c 中的函数即可。

虚拟地址空间

在 Lab 5 博客中我们提到用户进程的虚拟地址 0x60000000 下方有 4 MB 空间用于存储文件描述符 struct Fd，而上方有 32 * 4 MB 的空间来保存每个文件描述符对应的文件内容。对于控制台来说，它不需要 0x60000000 上方的空间，毕竟它不需要打开任何文件。

这是怎么一回事？

每个 struct Fd 内含了设备类型，因此在这 32 个文件描述符中有的是文件系统，有的是控制台。

每个文件描述符对应 0x60000000 上方的文件系统映射区中的一块，因此控制台和文件系统不会冲突。

接下来我们讲讲神奇的管道。

正如前面所说，管道也是一种数据结构，它是这样发挥效果的：

对于两个进程 readEnv 和 writeEnv（下面简称 RE 和 WE），虚拟地址 va1(RE) 和虚拟地址 va2(WE) 映射到同一个物理页面，并且该页面的权限位包含 PTE_LIBRARY。这样，当 WE 往页面写入内容的时候，RE 可以立刻感知到，而不用等待 IPC。

管道事实上是一种共享内存技术。

管道的结构体 struct Pipe 的定义如下：

写进程和读进程各有一个指针，表示读/写的位置，分别是 p_rpos 和 p_wpos。

另外我们有管道的缓冲区 p_buf。p_buf 看似特别小，只有 32 个 u_char 大小，实际上不是这样。

缓冲区实际上设计的是环形。即一个指针移动到头之后，会从另一头重新开始移动。

但我们为了简化设计，做了一些小调整：对于读指针和写指针，它们按理说增长到 31 就要返回 0 了。实际上，我们允许它们继续增长下去，这样更容易判断读写指针谁在前，谁在后。当然，在使用缓冲区的时候，还是需要对 32 取模的。

现在我们来考虑临界条件：读指针可以超过写指针吗？不行，否则就要读滚木了。因此读指针要落后于写指针。

读指针可以无限落后于写指针吗？也不行。因为这是个环形结构，倘若写指针把读指针「套圈」了，那么写指针写入的内容就会覆盖掉读指针还没读的内容。因此写指针超过读指针的距离不能超过一圈，即 32。因此我们得出了下面的临界条件：

先来理解一下如何判断读端或写端是否关闭。

我们用到 pageref 这个函数，该函数返回虚拟地址 va 对应物理页面的引用数 pp_ref。现在对于管道一侧的进程，它有 fd0 指向一个物理页（读端），fd1 指向一个物理页（写端），并且 data0 和 data1 共同指向一个物理页（管道）。如果记管道对应的虚拟地址为 pipe，我们有如下关系：

pageref(pipe) == pageref(fd0) + pageref(fd1)

那么，如果读或写的一端完全关闭，就有 pageref(pipe) == pageref(fd)，这里 fd 是 fd0 或 fd1。

读写竞争

管道是一个需要两个进程的机制。我们在创建管道的时候，其实做的事情是这样的：

  1. 调用 pipe 函数，为一个进程开启管道

  2. fork 该进程

我们称一个进程为 RE，另一个为 WE。假设先对 RE 开启管道，那么有：

  • fd0(RE) 指向页面 FD0

  • fd0(RE) 指向页面 FD1

  • data0(RE) 和 data1(RE) 指向页面 PIPE

在 fork 后，事情变得复杂起来了。不要忘记三个页面在分配时，权限位都标记了 PTE_LIBRARY。带有 PTE_LIBRARY 的页面与之前提到的写时复制 PTE_COW 一样，在 fork 时子进程的虚拟地址会和父进程的虚拟地址指向同一个物理页面。但与 PTE_COW 不同，带有 PTE_LIBRARY 的页面即使被写，也不会创建一个新的页面并将子进程的虚拟地址映射过去，它就是纯正的共享页面。也就是说 fork 后：

  • fd0(WE) 指向页面 FD0

  • fd0(WE) 指向页面 FD1

  • data0(WE) 和 data1(WE) 指向页面 PIPE

除此之外，由于读端和写端分离到两个进程中，对于 RE，我们应该关闭其写端 fd1(RE)；对于 WE，我们应该关闭其读端 fd0(WE)。

这时候你可能要问了：「那又怎么样呢？」

我们先来看读写端是如何关闭的：

该函数先取消 fd 的映射，再取消 data 的映射。为什么？可以反过来吗？

答案是不行，我们设想一种极端的情况：读进程 RE 已经关闭了写端，写进程 WE 正在关闭读端。

如果 WE 先关闭 data0(WE)，这时突然发生进程切换，切换到 RE 会怎么样？

  1. 由于 WE 还没有关闭读端，也就是 WE 根本没有准备好，它不可能往缓冲区写任何内容，写指针在 0 处

  2. 读进程尝试读取，但发现读指针和写指针都在 0 处，这时要检查写端（即写进程 WE）是否关闭

  3. 调用 _pipe_is_closed 检查页面 PIPE 和 FD0 的 pp_ref 是否相等

  4. 问题出现了：RE 的写端关闭了，因此 RE 中有一个虚拟地址指向 PIPE；WE 读端的 data0(WE) 取消映射了，因此也只有一个虚拟地址指向 PIPE。因此 PIPE 的 pp_ref 为 2。另一边 fd0(RE) 指向 FD0，但是 WE 的 fd0(WE) 还没来得及解除映射！因此 FD0 的 pp_ref 也等于 2

  5. RE 以为写端关闭了，故停止读取

也就是说，读进程 RE 还没有读取任何内容，就以为写进程 WE 已经写完了所有内容，直接停止读取了。这就是读写进程发生了同步问题。

但这样换一下顺序，能解决所有问题吗？答案是不能！

我们再设想一个极端的情况：读进程 RE 已经关闭了写端，写进程 WE 还没有关闭任何端。

这时假如读进程在读，碰到读写指针位置一致，开始检查写端是否关闭。如果刚执行完 fd_ref = pageref(fd)，就发生进程调度，轮到 WE 操作会怎么样？

  1. fd_ref == 2。

  2. WE 关闭 fd0(WE) 和 data0(WE)。

然后再发生进程调度，那么 RE 继续判断写端是否关闭，这时执行 pipe_ref = pageref(p) 得到 pipe_ref == 2，问题又出现了。

读进程又以为写端完全关闭了，于是不再读取任何内容。

总之，我们不能仅仅通过修改代码执行顺序来搞定这个竞争问题，我们需要更严苛的判断——在获取 pp_ref 的时候一定不能发生任何进程切换。这就要引出我们的 env_runs 了！

在初始化 PCB 时，我们将这个字段设置为 0。每次调用 env_run 函数的时候，我们将进程的 env_runs 自增 1。也就是说每次进程发生调度，该字段就会增加 1。

那么我们就可以利用这个字段来验证进程是否发生调度：我们将执行 pageref 之前该进程的 env_runs 记录下来。然后判断执行后的 env_runs 是否与之前的相等。如果不相等，就说明这中间发生了进程调度，计算结果不可信，必须重新计算；否则计算结果可信，用于推出最终的判断。

为什么？

用 ENV_CREATE 创建的进程在用 spawn 创建新进程后退出了。现在只剩这个新进程了，它申请的 struct Fd 必定是第一个。

接下来执行了 dup 这个函数，它定义在 user/lib/fd.c 中：

dup 的作用是这样的：对于一个进程，其 0x60000000 下方一个虚拟地址指向一个 struct Fd，并且对应 0x60000000 上方的虚拟地址指向数据（记作 data）。调用 dup 后，让一个新的虚拟地址指向 struct Fd，并让其对应的 0x60000000 上方虚拟地址指向 data。简单来说，就是复制一份映射关系。

提示

为什么要用 dup(0, 1)？在控制台一节我们提到，虽然控制台的 struct Fd 的打开模式是「读写」，我们还是约定 0 号用于读，1 号用于写。二者共用一个 struct Fd 也不会有任何问题，因为控制台不需要用到 offset 字段。

与此同时，这还代表 sh 是一条指令，作用是开启一个新的 Shell。它不用接收任何参数。

最后有一个 wait(r) 是什么意思？wait 用于检查进场 ID 为 r 的进程是否处于 ENV_FREE 状态，如果不是，则 syscall_yield。也就是说这个 wait 等的是刚创建的 Shell 执行：如果 Shell 进程结束了，就回到循环头再 spawn 一个新的 Shell 进程。

我们来查看 user/sh.c，从主函数开始：

如果输入的参数没有实现定义其行为，我们调用 usage 函数：

即提示使用方法并销毁进程。别忘了 Shell 进程被销毁后会立刻 spawn 一个新的。

接下来如果 argc > 1，也调用 usage。只有格式满足 sh [script-file] 我们才允许继续执行。

sh script-file 等价于 sh < script-file，之后讲到输入输出重定向时我们再来讨论这个问题。

总之，处理完了 sh 跟着的东西，我们就进入了 Shell 界面了。我们可以自由地在其中输入指令，因此 Shell 进程也需要进入死循环，时刻等待解析输入。

我们确保输入之前先输出一个 \n$ 来提示用户可以输入指令。然后调用 readline 读取一行输入。

该函数接收一个 buf 输入，以及要读取的字符数 n。在 n 轮循环中，每次都尝试读取一个字符。默认读取的来源是 fdnum 为 0 的 struct Fd，也就是功能为读的控制台。

如果读到 \b，即 ⌫，或者 0x7f，即 Delete，就反向移动 i 2 个位置。如果遇到换行符，就表示这一行输入结束，返回。

读取完一行输入后，我们可以判断它是否是注释，或者根据传给 sh 指令的参数来做一些别的事，但这都不是重点，重点是我们要执行这一行指令。

Shell 进程首先 fork 了一个子进程出来。然后子进程去执行 buf 的指令，执行完之后就销毁。父进程原地 wait 子进程被销毁。执行完毕后，回到循环开头再次接收输入。

来看 runcmd 函数：

gettoken 顾名思义是获取 s 的一个 token，但现在调用 gettoken(s, 0) 实际上只起到一个初始化的作用。具体原理可以自行参考源代码，这里不再赘述。

接着我们调用 parsecmd 函数，该函数有三个返回值，分别是 argc，argv 和 rightpipe。

函数总的来说就是不断的调用 gettoken 获取一个 token，然后根据其类型来更改 argc 和 argv。

我们以这样一条命令来举例：echo hello > helloWorld.txt

  1. 解析 echo，属于 'w' 类，将 argv[0] 填入 echo，break

  2. 回到循环开头，解析 hello，属于 'w' 类，将 argv[1] 填入 hello，break

  3. 回到循环开头，解析 >，属于 '>' 类，进入该 case。

  4. 再读一个 token，得到了重定向的文件 helloWorld.txt，类型必须是 'w'。

  5. 以 O_WRONLY | O_CREAT | O_TRUNC 打开该文件，这符合 Linux Shell 的情况。

  6. 调用 dup(fd, 1)，让 fdnum 为 1 的 struct Fd 映射到该文件。这里非常巧妙。本来我们要输出到屏幕，就需要调用 write(1, ...)，现在我们让 1 指向文件，今后调用 write 就重定向到了往文件中写入数据！

  7. 关闭 fd。

从刚才的例子我们也能参悟到输入重定向的实现方法：dup(fd, 0) 并修改打开模式为 O_RDONLY 即可。

  1. 回到循环开头，发现所有 token 都读完了，结束函数。

一会我们再来说说管道的实现，现在我们回到 runcmd 中。

在执行完 parsecmd 后，我们不要忘了在 argv 的最后加一个 0。

  1. 调用 spawn 函数，文件名是 echo，参数是 echo 和 hello。spawn 会交给 echo.b 去实现命令的效果。

  2. 命令执行完成之后，我们开始做善后工作：调用 close_all 关闭所有 struct Fd。并等待 spawn 出来的进程完成执行。

  3. 若 spawn 出来的进程已结束，则该进程也结束，回到 Shell 进程，Shell 进程进入下一个循环，等待输入下一条指令。

那么最后我们再来看看管道的情况。

如果 token 是 '|'，我们调用 pipe 创建一个管道，并返回两个 fdnum 到 p[2] 中。规定 p[0] 是读端，p[1] 是写端。

接着我们 fork 一个新进程，并将子进程 ID 写到 *rightpipe 中。接下来该进程执行的是 else 分支：将管道输出端 p[1] 给 dup 到 1 号 struct Fd 的位置，即控制台的输出。也就是说对于该进程的所有输出，都会被重定向到管道的写端。然后关闭 p[0] 和 p[1]。最后返回的是 | 左侧命令的 argc。

返回后，进程 spawn 一个新的进程去执行 | 左侧的命令。

对于之前 fork 出的新进程，情况也类似。它将 | 右侧的指令输入重定向到管道的读端。不同的是我们需要递归执行 parsecmd 去解析管道右侧的指令，并填写 argc 和 argv。

Thinking 6.3

若 PIPE_SIZE 过小，则首先更可能发生一次性传输数据大小超过 PIPE_SIZE 的情况，导致需要先对数据进行裁切，传输效率降低。其次读指针和写指针更可能抵达临界条件导致一方调用 syscall_yield 等待另一方，降低效率。

在适当的范围内提高缓冲区大小可以减少进程的切换，提高单次数据的传输量，进而提高效率和吞吐量。

Thinking 6.4

不是恒成立。比如在两个进程还没有来得及关闭自己不需要使用的端口时，该等式就不成立。当且仅当双方都关闭不需要的端口时，或者在此基础上一方关闭全部端口时才会成立。

不成立可能发生误判，以下摘自本人博客：

如果 WE 先关闭 data0(WE)，这时突然发生进程切换，切换到 RE 会怎么样？

  1. 由于 WE 还没有关闭读端，也就是 WE 根本没有准备好，它不可能往缓冲区写任何内容，写指针在 0 处

  2. 读进程尝试读取，但发现读指针和写指针都在 0 处，这时要检查写端（即写进程 WE）是否关闭

  3. 调用 _pipe_is_closed 检查页面 PIPE 和 FD0 的 pp_ref 是否相等

  4. 问题出现了：RE 的写端关闭了，因此 RE 中有一个虚拟地址指向 PIPE；WE 读端的 data0(WE) 取消映射了，因此也只有一个虚拟地址指向 PIPE。因此 PIPE 的 pp_ref 为 2。另一边 fd0(RE) 指向 FD0，但是 WE 的 fd0(WE) 还没来得及解除映射！因此 FD0 的 pp_ref 也等于 2

  5. RE 以为写端关闭了，故停止读取

也就是说，读进程 RE 还没有读取任何内容，就以为写进程 WE 已经写完了所有内容，直接停止读取了。这就是读写进程发生了同步问题。

Thinking 6.5

修改前的逻辑是先复制文件描述符的映射，再复制文件内容的映射。考虑子进程刚复制文件描述符 FD 的映射，发生进程切换。则此时 pageref(PIPE) == pageref(FD)，这导致父进程认为子进程已经关闭管道。

Thinking 6.6

系统调用是原子操作，其根本原因是避免同步问题。在进行系统调用时，已经关闭了中断，保证了系统调用的原子性。

  2. 会出现问题，正如 Thinking 6.5 所说。
